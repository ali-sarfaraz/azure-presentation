## Microsoft-hosted agent - On demand agents - Can take longer to provision
## Self Hosted Agents - Builds will run faster. - Leverage asset caching i.e. docker containers, yarn or composer. - Always running container.
## Mix And Match - We can use both microsoft-hosted agent and self hosted agent to do things in parallel. - Anything that doesn't required to be in our network can be carried on microsoft-hosted agent and anything that requires deployment can use self hosted agent.
## Multiple Self Hosted Agents Yes. This approach can work well for agents that run jobs that don't consume a lot of shared resources. For example, you could try it for agents that run releases that mostly orchestrate deployments and don't do a lot of work on the agent itself.
